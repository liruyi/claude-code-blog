AI Agent 的性能挑战与传统软件完全不同。API 延迟、Token 成本、并发瓶颈，每一个都是独特的优化对象。本系列从性能工程角度，分析 Claude Code 如何实现极致优化。

## AI Agent 的性能维度

传统软件性能关注：

| 维度 | 指标 |
|------|------|
| 响应时间 | 请求延迟 |
|吞吐量 | QPS |
| 资源使用 | CPU、内存 |

AI Agent 需要额外关注：

| 维度 | 指标 | 为什么重要 |
|------|------|------------|
| API 延迟 | 请求→首字节时间 | 用户等待体验 |
| Token 成本 | 输入+输出 Token | 每次调用都花钱 |
| 上下文管理 | 消息压缩效率 | 上下文有上限 |
| 工具并行度 | 同时执行工具数 | 工具执行是瓶颈 |

## 核心性能瓶颈

### 1. API 延迟（300ms-10s）

Claude API 响应不是瞬时的：

```
简单请求（100 tokens）：
- 首字节延迟：300ms
- 完成延迟：500ms

复杂请求（10,000 tokens）：
- 首字节延迟：2s
- 完成延迟：10s
```

**影响**：
- 用户感知延迟 = API 延迟 + 工具执行
- 长任务可能几分钟

### 2. Token 成本（$0.01-$1/请求）

Token 直接关联成本：

```
输入 10,000 tokens × $3/M = $0.03
输出 5,000 tokens × $15/M = $0.075

单次请求成本 ≈ $0.1
```

**影响**：
- 长对话成本累积
- 大上下文成本更高

### 3. 上下文上限（200K tokens）

Claude 有上下文限制：

```
最大上下文：200,000 tokens

实际可用：
- 系统提示：10,000 tokens
- 工具定义：5,000 tokens
- 用户消息：50,000 tokens（压缩前）
- AI 响应：30,000 tokens
- 工具输出：100,000 tokens（可能）

问题：工具输出很容易占满上下文
```

### 4. 工具执行时间（秒-分钟）

工具执行是最大延迟源：

```
Bash: 编译、测试 → 30s-2min
Read: 大文件 → 0.5s
WebFetch: 远程请求 → 2s-10s
```

## 性能优化策略概览

### 1. 延迟优化

```
启动时：
├─ 快速路径（跳过加载）
├─ 延迟导入（按需加载）
├─ API 预连接（减少 TLS 时间）

运行时：
├─ 流式响应（不等完成）
├─ 并行工具执行（同时多个）
├─ 推测执行（对话框时预先分类）
```

### 2. Token 优化

```
上下文管理：
├─ 系统提示压缩（保留关键）
├─ 工具输出摘要（不保留原始）
├─ 历史压缩（保留关键决策）
├─ 自动压缩（超限触发）

策略选择：
├─ Snip（快速剪裁）
├─ Microcompact（结构保留）
├─ Context Collapse（相似合并）
├─ Autocompact（阈值触发）
```

### 3. 并发优化

```
工具执行：
├─ 并发安全判断（哪些可并行）
├─ Streaming Tool Executor（边接收边执行）
├─ 批量执行（同类型工具）

子 Agent：
├─ Fork 模式（独立上下文）
├─ Context 克隆（共享读取状态）
```

### 4. 存储优化

```
会话存储：
├─ JSONL 格式（追加写入）
├─ 尾部读取（不读整个文件）
├─ Tombstone 限制（50MB）

文件缓存：
├─ Read 去重（已读文件不重复发送）
├─ mtime 追踪（文件修改时间）
```

## 性能原则预告

本系列将提炼 AI Agent 性能原则：

1. **流式优先原则**：永远不要等待完成再显示
2. **并行最大化原则**：能并行的一定并行
3. **Token 珍惜原则**：每个 Token 都要优化
4. **压缩智能原则**：压缩要保留关键信息
5. **缓存有效原则**：避免重复发送
6. **延迟隐藏原则**：预连接、推测执行
7. **上下文预算原则**：主动管理不超限
8. **增量处理原则**：不一次性处理全部
9. **隔离执行原则**：大任务用 Fork
10. **测量驱动原则**：优化要有数据支撑

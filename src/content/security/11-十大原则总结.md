本系列分析了 Claude Code 的安全防御体系。最后，我们提炼十大 AI Agent 安全原则。

## 原则 1：默认拒绝原则

**除非明确允许，否则拒绝**

### 安全默认值

```typescript
const TOOL_DEFAULTS = {
  isConcurrencySafe: () => false,  // 默认不安全
  isReadOnly: () => false,          // 默认写入
  checkPermissions: () => ({ behavior: 'ask' }),  // 默认询问
}
```

### 设计对策

```
工具必须显式声明：
- '我是安全的' → 才能并行执行
- '我是只读的' → 才能在 plan 模式执行
- '我通过检查' → 才能自动允许
```

### 实现要点

- 所有工具默认 `ask` 权限
- 危险操作必须有安全检查
- 用户必须显式允许

## 原则 2：多层防御原则

**单一检查点不可靠**

### 检查层次

```
第一层：工具自身检查
第二层：规则匹配
第三层：权限模式检查
第四层：AI 分类器
第五层：权限对话框
```

### 设计对策

```
每一层都可能阻止：
- 任一层 deny → 最终拒绝
- 所有层 allow → 执行
```

### 实现要点

- 不依赖单一检查
- 每层独立验证
- 检查结果可追溯

## 原则 3：最小权限原则

**工具只获得必要权限**

### 权限分类

```
Read：只读权限
Grep：只读权限
Edit：写入权限
Bash：执行权限（最高）
```

### 设计对策

```
Skill 可以限制工具：
allowedTools: [Read, Grep]  // 只能读，不能写
```

### 实现要点

- 工具声明自己的权限级别
- Skill 限制可用工具
- 模式限制执行范围

## 原则 4：透明决策原则

**安全决策要可解释**

### 决策透明

```
拒绝时说明原因：
'拒绝执行：rm -rf *'
'原因：递归删除所有文件是危险操作'
'分类器置信度：95%'
```

### 设计对策

```
✅ 透明决策
显示：
- 检查通过/失败
- 决策原因
- 用户可以覆盖
```

### 实现要点

- 每个检查返回原因
- 显示给用户
- 允许用户覆盖

## 原则 5：用户控制原则

**用户可以覆盖安全决策**

### 覆盖机制

```
用户可以：
- 允许一次（临时）
- 允许本次会话（会话规则）
- 永久允许（写入设置）
- 修改规则（调整策略）
```

### 设计对策

```
✅ 用户控制
安全拒绝后：
'您可以选择允许执行'
[允许一次] [允许本次会话] [永久允许]
```

### 实现要点

- 提供覆盖选项
- 不强制用户选择
- 记录用户决策

## 原则 6：审计记录原则

**所有关键操作要记录**

### 审计内容

```
记录：
- 工具名称
- 工具参数（敏感信息过滤）
- 执行时间
- 执行结果
- 用户决策（允许/拒绝）
```

### 设计对策

```
✅ 审计日志
~/.claude/audit.log:
2024-01-15T10:23:45 | Bash | git push | allowed | user_session
2024-01-15T10:24:00 | Edit | auth.ts | success
```

### 实现要点

- 关键操作写日志
- 过滤敏感信息
- 企业可集中收集

## 原则 7：隔离执行原则

**危险操作要沙箱化**

### 沙箱机制

```
沙箱隔离：
- 文件系统隔离
- 网络隔离
- 进程隔离
```

### 设计对策

```
✅ 沙箱执行
危险命令在沙箱中：
- 只能访问允许的目录
- 不能访问网络
- 不能修改系统文件
```

### 实现要点

- 使用 OS 沙箱技术
- 限制资源访问
- 提供沙箱选项

## 原则 8：AI 辅助原则

**用 AI 帮助安全决策**

### AI 分类器

```
AI 分类器：
- 分析命令语义
- 判断安全风险
- 提供决策建议
```

### 设计对策

```
✅ AI 辅助
AI 分类器建议：
'分类器判断：deny（置信度 95%）'
'原因：强制推送到主分支'
```

### 实现要点

- 使用快速模型（haiku）
- 高置信度自动决策
- 低置信度显示对话框

## 原则 9：规则优先原则

**用户规则优先于 AI 判断**

### 优先级

```
用户规则 > 企业规则 > AI 分类器 > 默认行为
```

### 设计对策

```
✅ 规则优先
用户设置：
{ 'Bash(git *)': 'allow' }

AI 分类器建议 deny：
但用户规则匹配 → 自动允许
```

### 实现要点

- 规则检查在分类器之前
- 用户规则最高优先
- 企业规则次优先

## 原则 10：持续更新原则

**威胁模型要持续更新**

### 威胁演进

```
新威胁：
- 新的攻击技术
- 新的工具类型
- 新的使用场景
```

### 设计对策

```
✅ 持续更新
定期更新：
- 危险模式列表
- 分类器训练
- 检查规则
```

### 实现要点

- 监控新攻击技术
- 更新安全规则
- 分类器重新训练

## 原则总结表

| 原则 | 核心问题 | 设计对策 |
|------|----------|----------|
| 1 默认拒绝 | 如何保证安全？ | 明确允许才执行 |
| 2 多层防御 | 如何避免单点失效？ | 五层检查 |
| 3 最小权限 | 如何限制风险？ | 工具权限分级 |
| 4 透明决策 | 如何让用户理解？ | 显示原因 |
| 5 用户控制 | 如何平衡安全与效率？ | 用户可覆盖 |
| 6 审计记录 | 如何追溯？ | 操作日志 |
| 7 隔离执行 | 如何防止破坏？ | 沙箱化 |
| 8 AI 辅助 | 如何智能判断？ | 分类器 |
| 9 规则优先 | 如何尊重用户？ | 规则优先 |
| 10 持续更新 | 如何保持安全？ | 定期更新 |

## 应用这些原则

当你设计 AI Agent 安全时：

1. **检查每一项**：是否满足所有原则？
2. **权衡取舍**：有时原则会冲突（如安全 vs 效率）
3. **威胁建模**：分析你的具体威胁
4. **迭代改进**：安全是持续的过程

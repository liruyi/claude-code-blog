本系列分析了 Claude Code 的用户体验设计。最后，我们提炼十大 AI CLI UX 设计原则。

## 原则 1：实时反馈原则

**永远不要让用户等待沉默**

### 传统 CLI vs AI CLI

```
传统 CLI：
命令 → 执行 → 输出 → 结束
（用户知道在做什么）

AI CLI：
意图 → 思考 → 可能多步 → 可能工具 → 最终响应
（用户不知道 AI 在做什么）
```

### 设计对策

```
❌ 空白等待
用户输入 ───────► [等待 30s] ───────► 输出

✅ 实时反馈
用户输入 ───► 显示思考动画 ───► 显示工具执行 ───► 流式输出 ───► 完成
           0.5s              5s              15s             30s
```

### 实现要点

- Spinner 动画表示正在处理
- 工具名称和参数摘要
- 进度百分比
- 流式文本输出

## 原则 2：信任建立原则

**每个操作都要解释原因**

### 用户焦虑

```
用户看到：
Claude 想要执行 rm -rf node_modules

用户担忧：
- 为什么要执行这个？
- 会删除什么？
- 安全吗？
```

### 设计对策

```
❌ 只显示命令
rm -rf node_modules

✅ 显示推理
Claude 想要清理项目依赖
因为：用户报告依赖版本冲突
影响：删除 500MB 文件
```

### 实现要点

- 显示 AI 的推理过程
- 说明操作影响范围
- 提供分类器建议

## 原则 3：用户控制原则

**用户随时可以干预**

### 干预时机

```
用户可以：
- 输入前：取消提交
- 执行中：Ctrl+C 中断
- 权限确认：拒绝操作
- 执行后：撤销（有限）
```

### 设计对策

```
✅ 明确的控制入口
- Ctrl+C 提示：'已暂停，使用 /resume 继续'
- 权限对话框：'拒绝'按钮明显
- 工具执行中显示：'可以按 Ctrl+C 中断'
```

### 实现要点

- AbortController 传递中断信号
- 中断不丢失已完成的工作
- 提供恢复机制

## 原则 4：信息精简原则

**终端空间有限，只显示关键信息**

### 信息层次

```
高优先级（必须显示）：
- 用户输入
- AI 响应核心内容
- 工具名称

中优先级（选择性显示）：
- 工具参数摘要
- 工具执行进度

低优先级（不默认显示）：
- 工具完整输出
- 调试日志
```

### 设计对策

```
❌ 显示所有
[详细日志 500 行]

✅ 精简显示
工具执行成功 [展开查看详情]
```

### 实现要点

- 默认折叠工具输出
- 提供展开入口
- 进度更新节流

## 原则 5：进度可见原则

**长任务必须有进度指示**

### 进度需求

```
用户等待时想知道：
- 正在做什么？
- 进度多少？
- 还要多久？
```

### 设计对策

```
✅ 进度显示
🔄 Bash: npm test ──────────────────────── 45%
         已执行 45/100 测试
```

### 实现要点

- 解析命令输出获取进度
- 显示百分比或已完成数量
- 更新频率 500ms

## 原则 6：中断友好原则

**Ctrl+C 不应该丢失工作**

### 中断处理

```
用户按 Ctrl+C：
- 保存当前进度
- 保存已完成的工作
- 提示恢复方法
```

### 设计对策

```
✅ 中断友好
^C
已暂停。已完成：
- 读取了 src/auth.ts
- 编辑了 src/api.ts

使用 /resume 继续或 /save 保存
```

### 实现要点

- AbortController 传递中断
- 会话状态持久化
- 提供恢复命令

## 原则 7：错误友好原则

**错误消息要可理解、可解决**

### 错误类型

```
用户输入错误 → 指导修正
工具执行失败 → 解释原因 + 建议
API 错误 → 简化提示
```

### 设计对策

```
❌ 技术性错误
Error: ENOENT: no such file or directory, open '/path/to/file'

✅ 用户友好错误
找不到文件：src/auth.ts

建议：
1. 检查文件路径是否正确
2. 使用 Glob 查找相似文件
```

### 实现要点

- 翻译技术错误为用户语言
- 提供解决建议
- 高亮关键信息

## 原则 8：引导渐进原则

**新用户不需要读文档**

### 渐进引导

```
第一次使用：
- 显示快速介绍
- 提示常用命令

第一次拒绝：
- 解释安全机制
- 提示如何设置规则

功能发现：
- 折叠提示新功能
- 适时推荐
```

### 设计对策

```
✅ 适时引导
当 Claude 想执行危险命令时：
'这是 Claude 第一次执行此类型命令。
您可以设置规则自动允许常见操作。'
```

### 实现要点

- 首次使用检测
- 适时功能提示
- 不过度打扰

## 原则 9：恢复无缝原则

**会话恢复感觉像继续**

### 恢复体验

```
用户期望：
恢复后 → 看到上次对话 → 继续聊天 → 不感觉中断
```

### 设计对策

```
✅ 无缝恢复
恢复会话...

上次对话：
用户: 重构 auth.ts
AI: 我来分析...
[已完成工具执行摘要]

继续对话：
> 继续优化 API 部分
```

### 实现要点

- 快速加载历史摘要
- 显示上次状态
- 不重新加载完整历史

## 原则 10：响应式原则

**适配不同终端大小**

### 终端适配

```
宽屏（> 120 cols）：
- 工具执行区域在侧边
- 状态栏更多信息

窄屏（< 80 cols）：
- 工具执行区域折叠
- 状态栏缩写
```

### 设计对策

```
✅ 响应式布局
检测终端宽度 → 调整布局

宽屏：侧边栏显示工具进度
窄屏：折叠工具进度，只显示状态
```

### 实现要点

- 监听终端大小变化
- 动态调整布局
- 关键信息不隐藏

## 原则总结表

| 原则 | 核心问题 | 设计对策 |
|------|----------|----------|
| 1 实时反馈 | 如何减少等待焦虑？ | Spinner + 工具进度 + 流式输出 |
| 2 信任建立 | 如何让用户信任 AI？ | 显示推理 + 影响说明 |
| 3 用户控制 | 如何让用户干预？ | Ctrl+C + 权限对话框 + 恢复 |
| 4 信息精简 | 如何节省空间？ | 折叠输出 + 摘要显示 |
| 5 进度可见 | 如何显示长任务？ | 百分比 + 已完成数量 |
| 6 中断友好 | 如何处理 Ctrl+C？ | 保存进度 + 恢复提示 |
| 7 错误友好 | 如何显示错误？ | 翻译 + 建议 + 高亮 |
| 8 引导渐进 | 如何帮助新用户？ | 首次提示 + 适时引导 |
| 9 恢复无缝 | 如何恢复会话？ | 摘要加载 + 状态显示 |
| 10 响应式 | 如何适配终端？ | 动态布局 + 关键保留 |

## 应用这些原则

当你设计 AI CLI 时：

1. **检查每一项**：是否满足所有原则？
2. **权衡取舍**：有时原则会冲突（如精简 vs 详细）
3. **用户测试**：实际用户会告诉你问题
4. **持续改进**：UX 是迭代的过程

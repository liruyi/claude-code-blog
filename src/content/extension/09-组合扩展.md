前三篇分别介绍了 Skills、Hooks、MCP。本篇展示如何组合三大机制，构建完整的自动化工作流。

## 为什么组合？

单一机制有局限：

| 机制 | 局限 |
|------|------|
| Skill | 只能定义流程，无法验证 |
| Hook | 只能响应事件，无法主动执行 |
| MCP | 只能提供工具，无法封装流程 |

组合使用可以：
- Skill 定义标准流程
- Hook 验证和记录
- MCP 提供底层工具

## 组合模式

### 模式 1：Skill + Hook 验证

**场景**：部署流程需要验证

```
用户: /deploy production
     │
     ▼
Skill 加载（deploy-skill.md）
     │
     ▼
PreToolUse Hook 验证：
     │
     ├─► 是 production 分支？
     │       │
     │       ├─► 否 → 阻止，提示错误
     │       │
     │       └─► 是 → 允许执行
     │               │
     │               ▼
     │           Claude 执行部署命令
     │               │
     │               ▼
     │           PostToolUse Hook：
     │               │
     │               ├─► 记录审计日志
     │               │
     │               └─► 发送 Slack 通知
```

### 实现

**Skill** (`deploy-skill.md`)：

```markdown
---
name: deploy
description: Deploy to specified environment
allowedTools: [Bash]
argumentHint: [environment: production|staging]
---

# Deploy Skill

Deploy the application to the specified environment.

## Steps

1. Run tests: `npm test`
2. Build: `npm run build`
3. Deploy:
   - production: `./scripts/deploy-prod.sh`
   - staging: `./scripts/deploy-staging.sh`
4. Verify deployment: `./scripts/health-check.sh`

Report success or failure to the user.
```

**Hook** (`settings.json`)：

```json
{
  'hooks': {
    'PreToolUse': [
      {
        'matcher': 'Bash',
        'hooks': [
          {
            'type': 'command',
            'command': '~/.claude/hooks/deploy-validator.sh'
          }
        ]
      }
    ],
    'PostToolUse': [
      {
        'matcher': 'Bash',
        'hooks': [
          {
            'type': 'command',
            'command': '~/.claude/hooks/deploy-notify.sh'
          }
        ]
      }
    ]
  }
}
```

**验证 Hook** (`deploy-validator.sh`)：

```bash
#!/bin/bash

input=$(cat)
command=$(echo '$input' | jq -r '.tool_input.command')

# 检查部署命令
if [[ '$command' =~ 'deploy-prod' ]]; then
  branch=$(git branch --show-current)

  if [[ '$branch' != 'production' ]]; then
    jq -n --arg branch '$branch' \
      '{continue: false, decision: 'block', reason: 'Production deployment requires production branch. Current: ' + $branch}'
    exit 0
  fi
fi

echo '{'continue': true}'
```

**通知 Hook** (`deploy-notify.sh`)：

```bash
#!/bin/bash

input=$(cat)
command=$(echo '$input' | jq -r '.tool_input.command')

# 部署命令后发送通知
if [[ '$command' =~ 'deploy-' ]]; then
  curl -s -X POST '$SLACK_WEBHOOK_URL' \
    -H 'Content-type: application/json' \
    -d '{'text': '🚀 Deployment initiated by Claude Code'}'
fi

echo '{'continue': true}'
```

### 模式 2：MCP + Skill 封装

**场景**：Slack 消息发送流程

Skill 封装 MCP 工具调用，简化用户操作。

**MCP Server**（Slack）提供工具：
- `slack_post_message`
- `slack_get_messages`
- `slack_search`

**Skill** (`slack-notify.md`)：

```markdown
---
name: slack-notify
description: Send a notification to Slack channel
allowedTools: [mcp__slack__slack_post_message]
argumentHint: [channel: message]
---

# Slack Notify Skill

Send a notification to the specified Slack channel.

## Input Format

The argument should be: `<channel>: <message>`

Example: `engineering: Build completed successfully`

## Steps

1. Parse the channel and message from the argument
2. Call `mcp__slack__slack_post_message` with:
   - channel: extracted channel name
   - text: extracted message
3. Confirm message was sent
```

**使用**：

```
/slack-notify engineering: Bug fixed, tests passing
```

**效果**：
- Skill 解析参数
- Skill 调用 MCP 工具
- MCP 工具执行 Slack API

### 模式 3：Hook 验证 MCP 工具

**场景**：限制 MCP 工具的使用范围

Hook 在 MCP 工具执行前验证。

**Hook** (`settings.json`)：

```json
{
  'hooks': {
    'PreToolUse': [
      {
        'matcher': 'mcp__slack__*',
        'hooks': [
          {
            'type': 'command',
            'command': '~/.claude/hooks/slack-guard.sh'
          }
        ]
      }
    ]
  }
}
```

**验证 Hook** (`slack-guard.sh`)：

```bash
#!/bin/bash

input=$(cat)
tool_name=$(echo '$input' | jq -r '.tool_name')
channel=$(echo '$input' | jq -r '.tool_input.channel // empty')

# 只允许发送到特定频道
allowed_channels=('engineering' 'alerts' 'support')

if [[ -n '$channel' ]]; then
  for allowed in '${allowed_channels[@]}'; do
    if [[ '$channel' == '$allowed' ]]; then
      echo '{'continue': true}'
      exit 0
    fi
  done

  jq -n --arg channel '$channel' \
    '{continue: false, decision: 'block', reason: 'Channel ' + $channel + ' not allowed. Allowed: engineering, alerts, support'}'
  exit 0
fi

echo '{'continue': true}'
```

**效果**：
- Claude 尝试发送 Slack 消息
- Hook 检查频道是否允许
- 不允许则阻止

### 模式 4：完整审批流程

**场景**：PR 审批自动化

结合 Skill、Hook、MCP 实现完整流程。

**流程图**：

```
用户: /approve-pr 123
     │
     ▼
Skill: approve-pr
     │
     ▼
PreToolUse Hook:
     │
     ├─► 检查 PR 状态
     │       │
     │       ├─► Draft → 阻止
     │       │
     │       └─► Ready → 允许
     │               │
     │               ▼
     │           MCP: GitHub
     │               │
     │               ├─► 获取 PR 详情
     │               │
     │               ├─► 检查 CI 状态
     │               │
     │               └─► 添加 approval review
     │                       │
     │                       ▼
     │                   PostToolUse Hook:
     │                       │
     │                       ├─► 记录审计日志
     │                       │
     │                       └─► 发送 Slack 通知
```

**实现**

**Skill** (`approve-pr.md`)：

```markdown
---
name: approve-pr
description: Approve a GitHub PR after validation
allowedTools: [mcp__github__*, Bash]
argumentHint: [PR number]
---

# Approve PR Skill

Approve the specified PR after checking CI status.

## Steps

1. Get PR details: `mcp__github__get_pull_request`
2. Check CI status: `mcp__github__get_pull_request_reviews`
3. If CI passed and not draft:
   - Add approval review: `mcp__github__create_pull_request_review`
4. Report result to user
```

**PreToolUse Hook**：

```bash
#!/bin/bash
# pr-validator.sh

input=$(cat)
tool_name=$(echo '$input' | jq -r '.tool_name')

# 检查 GitHub 工具调用
if [[ '$tool_name' =~ 'mcp__github__create_pull_request_review' ]]; then
  pr_number=$(echo '$input' | jq -r '.tool_input.pull_number')

  # 获取 PR 状态（需要 MCP GitHub 已连接）
  pr_status=$(curl -s '$GITHUB_API_URL/pulls/$pr_number' | jq -r '.draft')

  if [[ '$pr_status' == 'true' ]]; then
    jq -n --arg pr '$pr_number' \
      '{continue: false, decision: 'block', reason: 'PR #' + $pr + ' is draft, cannot approve'}'
    exit 0
  fi
fi

echo '{'continue': true}'
```

**PostToolUse Hook**：

```bash
#!/bin/bash
# pr-notify.sh

input=$(cat)
tool_name=$(echo '$input' | jq -r '.tool_name')

if [[ '$tool_name' =~ 'mcp__github__create_pull_request_review' ]]; then
  pr_number=$(echo '$input' | jq -r '.tool_input.pull_number')

  # 发送 Slack 通知
  curl -s -X POST '$SLACK_WEBHOOK_URL' \
    -H 'Content-type: application/json' \
    -d '$(jq -n --arg pr '$pr_number' '{text: '✅ PR #' + $pr + ' approved'}')'
fi

echo '{'continue': true}'
```

## 组合最佳实践

### 1. 分层职责

| 机制 | 职责 |
|------|------|
| Skill | 流程编排、用户交互 |
| Hook | 验证、审计、通知 |
| MCP | 底层工具、外部系统 |

### 2. 优先级

Hook 先于 Skill 执行：
- PreToolUse Hook 可以阻止 Skill 发起的工具调用
- Skill 无法绕过 Hook

### 3. 错误处理

各层独立处理错误：
- MCP 工具失败 → Skill 处理
- Hook 阻止 → Skill 收到阻止消息

### 4. 可测试性

分层测试：
- MCP Server 单独测试（协议测试）
- Skill 单独测试（提示测试）
- Hook 单独测试（验证测试）
